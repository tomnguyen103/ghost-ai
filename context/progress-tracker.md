## Architecture Decisions (slug-based URLs + room ID alignment)

- `project.id = project.slug = roomId` — the client generates `toSlug(name) + "-" + randomSuffix` and sends it as `id` in the create request. The API stores it as both `id` and `slug` in a single query. This keeps the DB primary key, the URL slug, and the Liveblocks room ID as one value throughout the system.
- The 6-char random suffix is generated when the create dialog opens (not on submit), so the preview in the dialog is stable while the user types the name.
- `CanvasRoom` receives `project.slug` as its `roomId`. The Liveblocks auth route calls `getProjectAccess(room, ...)` which looks up by `slug`, so the room ID and the lookup key always match.
- `slug` does not change on rename — only `name` is updated. Shareable links and Liveblocks room history are preserved across renames.
- `getProjectAccess` looks up exclusively by `slug` (`findUnique({ where: { slug } })`). Since `id = slug` for all newly created projects, any old UUID-based projects would need to be recreated to work with the new routing.
